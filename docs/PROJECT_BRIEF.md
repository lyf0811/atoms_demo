# Atoms Demo 项目简介

## 1. 实现思路与关键取舍
由于当前做的是一个demo版本且时间有限，因此架构需要简化，所有涉及生产环境中的场景都需要寻找简单的代替，能实现功能即可。

### 1.1 总体实现思路



- 整体采用 Next.js 全栈应用加自定义 Node server 的方式实现一个轻量级的demo。Next.js 负责页面、API、认证和业务接口；自定义 Node server（`server.mjs`） 负责 Next.js 默认能力之外的实时能力

- 全程涉及的相关数据都存储在本地，比如用户账户、和大模型的交互对话、大模型生成的代码，产品市场的项目等。

- 全程用到的服务器资源也使用本机资源，比如agent、项目的预览，数据的读取存储等

- 最核心的与agent的交互思路：是在web端打开了服务器的一个终端，执行已有的AI编码工具CLI（此处用的是opencode cli），同时接通用户的输入，监听agent的输入输出，最终实现了Chat交互。

- 延伸能力：看到atoms是有通过模版开始一个项目的能力的，但是仅提供基础的功能，比较粗糙，所以想要做一个内部市场，用户可以将自己成熟度高的项目发布到上面提供给别的用户使用（这里面可以免费也可以收费——保护知识产权）。然后其他用户应用了这个项目就会把对应的chat内容，代码等都复制过来。


### 1.2 关键取舍

**取舍一：应该存放在数据库里的数据，都放在了本地**

考虑到是demo版本，外接数据库做一系列的分层会很耗时且消耗服务器资源，所以将数据，比如用户的账户密码，和大模型的会话，项目的代码，应用市场的项目这些数据都存放在本地，对于demo的演示已经足够了。

**取舍二：使用现有的AI编程工具CLI模拟替代了真实应用级的agent**

实际要做一个完整的产品，应该是要做出一个专门的产品agent，这是一个非常大型的工程，涉及很多模块，很多步骤（提示词，上下文，智能体循环，记忆，各种工具，安全，各种协议等等）。对于demo来说，使用一个现有的AI编程工具来模拟已经满足了基本能力，可以根据用户的输入输出来提供对应的代码。

**取舍三：生成的项目部署在本地进行预览而不是真实部署到服务器上**

在初始化一个项目的时候会预制一套环境模版，agent会基于这个环境模版和用户的输入来生成项目，预览的时候项目就在本地服务器上新开一个端口进行通信，相当于部署在了本地，外界能预览也能单独访问。

## 3. 后续扩展路线与优先级

如果后续继续投入时间，以安全性和稳定性优先，然后再做生产化（从一个真实的idea到产品的流程出发）。

### P0：稳定性与基础安全

如果仍然是单机场景，当前优先级最高的是文档和安全，因为当前系统已经接入真实 terminal 和 npm 执行，稳定性和边界控制必须先做好。

后续工作：
- 引入 Docker 或独立 runner 隔离每个项目的 preview。
- 给 preview 设置 CPU、内存、执行时长限制。
- 限制 opencode 和 terminal 只能访问当前项目目录。
- 增加进程管理面板，能查看并停止用户 terminal/preview。
- 增加任务取消机制。
- 增加更可靠的端口分配。

这些问题不解决，系统很难安全地开放给外部用户。尤其是 preview 执行 npm 命令，必须尽早沙箱化。

### P1：数据层与项目持久化升级

下一步应把数据由本地存储转换成正式数据层，将服务和数据存储分离。

后续工作：
- 用户、项目、市场、会话迁移到数据库（例如 PostgreSQL）。
- session 迁移到 Redis 或数据库 session。
- code snapshot 和 market package 存到对象存储。
- Chat 记录结构化存储。
- 增加项目版本表。
- 增加数据库迁移脚本。

demo版本可以将数据保存在本地，但随着项目扩展以及有真正的用户使用，会出现各种数据读写机及安全性问题。数据层升级是继续扩展的基础。

### P2：真实部署能力

当数据和安全性得到保证后，需要关注的是部署环节，要能够独立出来，真实的部署到生产机器上。

后续工作：
- 将服务部署在真实的生产服务器上（例如 Vercel / Cloudflare Pages / 自建部署 worker）。
- 每个项目生成独立 deploy URL、完成对应的域名备案。
- 发布前执行 build/test。
- 保存部署历史。
- 支持回滚。
- Market 项目可以展示 live demo。

一旦有了真正的用户，就意味着我们做的是真正的产品，此时必须将用户的项目与服务分离，部署到真正的服务器上，以用户划分独享资源。

### P3：构建真实Agent

目前的Agent交互由AI Coding工具模拟实现，在这之中，各方面已经被限制，如果产品要真正的可用、好用，则需要以目标产品为导向开发真实的Agent。

后续工作：
- 完整的控制系统（执行循环、规划、推理、状态管理、记忆系统、工具集成……）
- 完整的安全性管控（沙箱、权限管理……）
- 多Agent协作
- 可观测性（结构化日志、指标监控、追踪和调试）
- 完整的评估能力。
- 可持续化迭代和工程能力

拥有了独属于产品的Agent，才会有质的飞跃，才能着眼于整个项目，而不是AI Coding工具的模拟。

### P4：测试、CI/CD 与质量体系

当产品达到一定规模时，测试、CI/CD 与质量体系这套经典系统就派上了作用，能够保障产品质量，减少出问题的概率。

后续工作：
- 增加组件测试：Projects、Chat、Code、Market。
- 增加 API 测试：auth、projects、market、workspace。
- 构建产品CI/CD流水线。
- 添加质量门禁（代码检查、集成测试、评估阈值）。

大型项目都会有对应的质量看护体系，不然随着代码的发展，用户的增多，问题会更集中的爆破。

### P5：产品体验能力升级及商业化
当拥有了较多的用户和稳定的用户数量增长后，就要提升产品的体验，增加用户的粘性，以及扩展产品的商业化（盈利能力）

后续工作：
- 体验升级（UI界面优化、产品流程简化……）
- 能力扩展（不仅限于Web端网页产品、不局限技术栈）
- 应用市场（打造成一个可开源、可付费、可参考的专属应用市场）。
- 为用户的创业项目添加营收点（比如现在已有的SEO和广告，可以扩展成GEO，以及付费软件等）

这些功能能够增加产品的价值，但在初期不具备高优先级。
